# Implementation Plan: Fraud Detection System for E-commerce

**Branch**: `001-fraud-detection-system` | **Date**: 2026-03-13 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `/specs/001-fraud-detection-system/spec.md`

## Summary

Build a **portfolio-grade, fully functional fraud detection system** for ecommerce platforms. The system detects fraudulent transactions in real-time using machine learning, provides explainability for every prediction, maintains comprehensive audit logs, displays analytics via a visual dashboard, and integrates with WooCommerce via a PHP plugin.

**Technical Approach**: Python 3.11 + FastAPI backend with async SQLAlchemy, PostgreSQL 16 database, scikit-learn RandomForest model with SHAP explainability, React 18 + Vite frontend dashboard, and PHP 8.x WooCommerce plugin. Deployment via Docker Compose for single-machine demonstration. Target: 5-minute setup from `git clone` to live prediction.

## Technical Context

**Language/Version**: Python 3.11 (backend), JavaScript/TypeScript (frontend), PHP 8.x (plugin)
**Primary Dependencies**: FastAPI ≥0.110, SQLAlchemy ≥2.0.25 (async), scikit-learn ≥1.4, SHAP ≥0.44, React 18.x, Recharts 2.x
**Storage**: PostgreSQL 16 with JSONB columns for audit/feature data, asyncpg driver
**Testing**: pytest (backend), Vitest (frontend), manual integration testing (plugin)

**Constraints**: Hard latency cap 500ms, recall ≥0.90, FPR ≤0.05, precision ≥0.85, single-machine deployment
**Scale/Scope**: Portfolio demonstration scope — 5 user stories, 15 functional requirements, 54 tasks across 8 phases

## Architecture Decisions

### 1. Machine Learning Model

**Decision**: scikit-learn RandomForest (n_estimators=200, class_weight='balanced', random_state=42)

**Rationale**:
- Fast training (<10s on 100k samples) and inference (<5ms per prediction)
- TreeExplainer from SHAP provides exact feature contributions in <5ms
- No GPU required — runs on any server
- Handles class imbalance via `class_weight='balanced'`

**Alternatives Considered**:
- **XGBoost/LightGBM**: Slightly better accuracy but slower SHAP explanations (10-20ms)
- **Neural Networks (PyTorch)**: Requires GPU, harder to explain, over-engineered for tabular data

**Tradeoff**: RandomForest is ~2-3% less accurate than gradient boosting but meets all metric gates and latency requirements.

### 2. Backend Framework

**Decision**: FastAPI with async SQLAlchemy

**Rationale**:
- Async-first design enables non-blocking DB writes (predictions logged without blocking response)
- Automatic OpenAPI docs for API exploration
- Pydantic validation prevents malformed input from reaching model
- Best Python ML ecosystem integration

**Alternatives Considered**:
- **Django REST**: Sync-only, slower, heavier
- **Flask**: No built-in async, manual validation
- **Node.js Express**: Poor ML library support

### 3. Database

**Decision**: PostgreSQL 16 with JSONB columns

**Rationale**:
- JSONB for flexible audit/feature storage (no schema migration for new features)
- Native UUID support for transaction IDs
- Production-grade with asyncpg driver (10x faster than psycopg2)
- Easy Docker deployment

**Alternatives Considered**:
- **SQLite**: No async support, single-writer bottleneck
- **MongoDB**: No strong consistency, overkill for structured data

### 4. Explainability

**Decision**: SHAP TreeExplainer with cached explainer object

**Rationale**:
- Tree models get exact Shapley values (not approximations)
- Preloading explainer at startup amortizes cost (<5ms per prediction)
- Industry-standard method, meets GDPR Article 22 requirements

**Alternatives Considered**:
- **LIME**: Approximation-based, slower, less accurate
- **Feature importance from model**: Not instance-specific

### 5. Rate Limiting

**Decision**: In-memory token bucket (100 req/s per client)

**Rationale**:
- Simple, no external dependencies
- Sufficient for single-machine demo
- Prevents adversarial probing

**Alternatives Considered**:
- **Redis**: Over-engineered for portfolio scope
- **None**: Exposes system to abuse

### 6. Fallback Strategy

**Decision**: Allow <$50 transactions, queue rest for manual review

**Rationale**:
- Prevents complete service outage from blocking all sales
- Low-value transactions have lower fraud risk
- Configurable threshold per merchant

**Alternatives Considered**:
- **Fail-closed (block all)**: Unacceptable business impact
- **Allow all**: Defeats purpose of fraud detection

### 7. Deployment

**Decision**: Docker Compose (3 services: postgres, backend, frontend)

**Rationale**:
- Single `docker compose up` for reviewers
- Meets "5-minute setup" portfolio goal
- Sufficient for single-machine demo

**Alternatives Considered**:
- **Kubernetes**: Over-engineered for demo
- **Manual setup**: Too complex for reviewers
